# AGENTS.md

このワークスペースは、C# / .NET を環境構築から実務レベルまで学ぶための Markdown ナレッジベースです。AI エージェントは、記事を単体で追加・修正するだけでなく、索引、ロードマップ、用語集、更新履歴との整合性を必ず保ってください。

## 基本方針

- このファイルはワークスペース全体に適用される agent instructions です。
- AGENTS.md は標準 Markdown として書き、必要なルールだけを簡潔に保ちます。
- 既存の記事構成、見出し、文体、ファイル命名に合わせます。
- 新しい内容は、初学者が読める説明と、実務での使い方・よくあるミスを両方含めます。
- 公式ドキュメントや一次情報へのリンクを優先します。

## ドキュメント更新時の必須ルール

Markdown 記事を追加・改名・削除・大きく更新した場合は、関連ファイルも同時に更新してください。

- `docs/index_page.md`: 新規記事、削除記事、移動記事のリンクを必ず反映する。
- `docs/roadmap.md`: 学習順、到達目標、実践課題に影響する変更があれば更新する。
- `docs/glossary.md`: 新しい重要用語、略語、実務用語を追加した場合は追記する。
- `docs/changelog.md`: 構成変更、主要記事追加、カテゴリ単位の更新を記録する。
- `mkdocs.yml`: サイトのナビゲーションに影響するカテゴリや主要ページを追加・改名・削除した場合は更新する。
- `docs/` 配下の各カテゴリの `README.md`: カテゴリ内の記事一覧、到達目標、次に進むカテゴリを最新化する。
- `docs/90_逆引き/`: エラー、ハマりどころ、コマンド、調査手順が増えた場合は逆引き記事にも反映する。
- `docs/80_演習問題/README.md`: 記事ごとの練習問題、演習問題、発展課題を追加・更新した場合はここに集約し、各記事本文には演習問題セクションを置かない。
- `docs/99_参考資料/`: 公式リンク、バージョン差分、学習資料に関係する変更があれば反映する。
- `作業/レビュー/`: レビュー結果、対応待ちの指摘、確認待ち事項を保存する。
- `作業/アーカイブ/`: レビュー指摘へ対応し、検証が終わった記録を移動する。
- `作業/ファクトチェック/`: ファイルごとのファクトチェック日時、根拠、結果を記録する。

## レビューとファクトチェック運用

- レビューした結果は `作業/レビュー/YYYY-MM-DD_対象_レビュー.md` に保存します。
- レビュー指摘へ対応し、必要な検証が終わったら `作業/アーカイブ/` に移動します。
- ファクトチェックはファイル単位で行い、`作業/ファクトチェック/台帳.md` に最終確認日時、確認者、根拠、結果を記録します。
- 公式リンク、インストール手順、バージョン差分、コマンド、セキュリティ注意点を更新した場合は、優先してファクトチェック対象にします。

## 記事構成

通常記事は、可能な限り次の構成に揃えます。

- 目的
- 前提
- 要点
- コード例またはコマンド例
- コードの読み方
- 実務での使い方
- よくあるミス
- 確認観点
- 関連リンク

実習記事は、可能な限り次の構成に揃えます。

- 目的
- 作るもの
- 学習ポイント
- 手順または確認コマンド
- コードの読み方
- 完了条件

テンプレートが必要な場合は、[docs/_templates/article.md](docs/_templates/article.md) と [docs/_templates/practice.md](docs/_templates/practice.md) を参照してください。

## ファイル命名と配置

- 公開する記事は `docs/` 配下の該当カテゴリに置きます。
- カテゴリ内の記事は `01_テーマ.md` のように番号順にします。
- 既存の日本語ファイル名と英数字ファイル名の使い分けを尊重します。
- 改名した場合は、参照元の Markdown リンクをすべて更新します。
- 画像や図は `docs/_assets/`、短い再利用コード片は `docs/_snippets/` に置きます。
- レビュー、ファクトチェック、アーカイブなどの管理記録は `作業/` に置き、MkDocs のナビゲーションには含めません。

## C# / .NET 記事の内容方針

- Windows / PowerShell / VS Code / C# Dev Kit を主環境として説明します。
- コマンド例は PowerShell でそのまま試せる形を優先します。
- .NET 10 LTS を主軸にし、必要に応じて .NET 8 LTS との差分を明記します。
- C# の文法だけでなく、テスト、DI、ログ、DB、Web API、CI/CD、運用観点まで実務に接続します。
- コード例は小さく、読みやすく、意図が分かるものにします。
- C# のコード例には、初学者が処理の流れを追える範囲で適宜 `//` コメントを入れます。特に、入力、処理、出力、例外処理、DI 登録、非同期処理、DB 操作、セキュリティ上の注意はコメントで補足します。
- コード例を載せる記事では、原則として `コードの読み方` セクションを追加し、重要な行、処理の順番、設計意図、実務での置き換え方、よくある誤読を説明します。
- コメントは行の意味をなぞるだけにせず、「なぜこの API を使うのか」「なぜこの順番なのか」「どこを実務では差し替えるのか」が分かる内容にします。
- セキュリティ、秘密情報、個人情報、認証認可、ログ出力に関係する内容では実務上の注意点を明記します。

## 検証ルール

Markdown を更新したら、可能な範囲で次を確認してください。

- VS Code 診断に Markdown lint やリンク関連のエラーがないこと。
- ローカル Markdown リンクが切れていないこと。
- `docs/index_page.md` から新規記事へ到達できること。
- カテゴリ `README.md` の記事一覧と実ファイルが一致していること。
- MkDocs 関連の構成を変更した場合は、ローカル Python ではなく Docker で `docker compose run --rm docs mkdocs build --clean` が成功すること。
- `今後追加する記事` のような未実体リストを残す場合は、意図的な backlog であることが分かるようにする。

Windows PowerShell 5.1 で Markdown を検証する場合、日本語パスの文字化けを避けるため `Get-Content -Encoding UTF8` を明示してください。
