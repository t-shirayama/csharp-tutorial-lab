# 境界値と例外のテスト

## 目的

境界値、無効値、例外を xUnit で検証し、仕様の抜けを見つけやすいテストを書けるようにします。

## 前提

- [xUnit の基本](01_xUnitの基本.md) を読んでいる
- [Arrange / Act / Assert](02_ArrangeActAssert.md) を読んでいる
- [Validation](../10_WebとAPI/06_Validation.md) を読んでいる

## 要点

- 境界値は不具合が出やすい値です。`0`、`1`、最大値、最小値、空文字、null、上限ちょうど、上限超えを優先して確認します。
- `[Theory]` と `[InlineData]` は、同じ仕様を複数の入力で検証するときに使います。テストコードの重複を減らしつつ、仕様の範囲を読みやすくできます。
- 例外を検証するときは `Assert.Throws` または `Assert.ThrowsAsync` を使います。例外が起きることだけでなく、例外の型と必要なら parameter 名や message も確認します。
- 例外を投げるべき失敗と、`false` や `Result<T>` で返すべき失敗は分けて考えます。入力ミスのような通常ケースをすべて例外にすると、呼び出し側が扱いにくくなります。
- テスト名には、対象、条件、期待結果を書きます。境界値テストでは、どの値が境界なのかが名前や `InlineData` から読めるようにします。
- 1つのテストに境界値を詰め込みすぎると、失敗時にどの条件が壊れたか分かりにくくなります。同じルールの複数値は `Theory`、別ルールは別テストに分けます。

## テスト対象の例

```csharp
public sealed class DiscountCalculator
{
    public decimal ApplyDiscount(decimal price, decimal discountRate)
    {
        if (price < 0)
        {
            throw new ArgumentOutOfRangeException(nameof(price));
        }

        if (discountRate < 0 || discountRate > 1)
        {
            throw new ArgumentOutOfRangeException(nameof(discountRate));
        }

        return price * (1 - discountRate);
    }
}
```

## 境界値を Theory で検証する

```csharp
public class DiscountCalculatorTests
{
    [Theory]
    [InlineData(1000, 0, 1000)]
    [InlineData(1000, 0.1, 900)]
    [InlineData(1000, 1, 0)]
    public void ApplyDiscount_ReturnsDiscountedPrice(
        decimal price,
        decimal discountRate,
        decimal expected)
    {
        var calculator = new DiscountCalculator();

        var result = calculator.ApplyDiscount(price, discountRate);

        Assert.Equal(expected, result);
    }
}
```

## 例外を検証する

```csharp
public class DiscountCalculatorExceptionTests
{
    [Theory]
    [InlineData(-1, 0)]
    [InlineData(1000, -0.01)]
    [InlineData(1000, 1.01)]
    public void ApplyDiscount_Throws_WhenInputIsOutOfRange(decimal price, decimal discountRate)
    {
        var calculator = new DiscountCalculator();

        Assert.Throws<ArgumentOutOfRangeException>(() =>
            calculator.ApplyDiscount(price, discountRate));
    }
}
```

## parameter 名を確認する

```csharp
[Fact]
public void ApplyDiscount_ThrowsPriceException_WhenPriceIsNegative()
{
    var calculator = new DiscountCalculator();

    var exception = Assert.Throws<ArgumentOutOfRangeException>(() =>
        calculator.ApplyDiscount(-1, 0.1m));

    Assert.Equal("price", exception.ParamName);
}
```

## コードの読み方

`InlineData` は、入力値と期待値の組み合わせです。最初のテストでは、割引率 `0`、通常の割引、割引率 `1` という境界を確認しています。例外テストでは、無効な値を渡したときに `ArgumentOutOfRangeException` が出ることを検証しています。parameter 名まで確認すると、呼び出し側が原因を特定しやすい例外になっているかも確認できます。

## 実務での使い方

金額、数量、日付範囲、文字列長、ページサイズ、権限、在庫数などは境界値が重要です。仕様書や validation rule を読んだら、正常値だけでなく「ちょうど境界」「1つ下」「1つ上」をテストにします。Web API では、同じ観点を status code や `ProblemDetails` の検証にもつなげます。

## よくあるミス

- 正常系だけテストして、0件、null、空文字、上限超えを確認しない。
- 例外の型を確認せず、どんな例外でも通るテストにする。
- `Theory` に無関係な仕様を詰め込み、テスト名と入力値の意味がずれる。
- 例外 message の全文に依存し、文言変更だけで壊れるテストにする。

## 練習問題

1. 数量が `1` 以上であることを、`0`、`1`、`2` でテストする。
2. 文字列長の上限ちょうどと上限超えをテストする。
3. `ArgumentOutOfRangeException` の parameter 名を確認する。
4. Web API の validation error を `400` と `ProblemDetails` で確認する。
