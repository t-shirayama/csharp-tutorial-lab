# コードレビュー観点

## 目的

実務でレビューするときに見るべき観点を整理します。

## 観点

- 要件を満たしているか。
- エラー時の挙動が明確か。
- null、空、境界値を扱っているか。
- 責務が適切に分かれているか。
- テストで重要な仕様が押さえられているか。
- ログや設定値が運用に耐えるか。
- 秘密情報や個人情報を漏らしていないか。

## テストと品質の観点

- 変更された仕様に対応するテストが追加・更新されているか。
- `dotnet test` がローカルと CI の両方で通るか。
- カバレッジの数字だけでなく、重要な分岐がテストされているか。
- 境界値、例外、非同期、キャンセル、テストデータの独立性が確認されているか。
- 外部依存のモックが過剰で、実装詳細に依存していないか。
- DB や外部 API を含むテストは、実行条件とデータ初期化が明確か。

## 実務での使い方

レビューは好みの指摘ではなく、バグ、保守性、運用リスクを下げるために行います。命名や整形は自動化し、設計と仕様に集中します。

PR でそのまま使える checklist と CI gate の例は [品質ゲートと PR チェック](18_品質ゲートとPRチェック.md) で扱います。

## よくあるミス

- スタイル指摘だけで終わる。
- 仕様確認をせず実装だけ見る。
- 大きすぎる PR をそのままレビューする。
- カバレッジ率だけを見て、テストの質を見ない。
- テストが flaky でも放置する。
