# テストデータ管理

## 目的

テストで使う入力データや期待値を整理し、読みやすく壊れにくいテストを書けるようにします。

## 前提

- [Arrange / Act / Assert](02_ArrangeActAssert.md) を読んでいる
- [テストしやすい設計](04_テストしやすい設計.md) を読んでいる

## 要点

- テストデータは仕様を表します。どの値が重要で、どの値はただの既定値なのかが読めるようにします。
- 各テストで大量の object 初期化を書くと、Arrange が長くなり、何を検証したいのか見えにくくなります。
- test data builder は、既定値を持つ object を作り、テストごとに重要な値だけ上書きする方法です。
- object mother は代表的な object を返す helper です。便利ですが、共有データに依存しすぎると変更の影響が広がります。
- fixture は重い初期化や共通 resource を共有できます。ただし共有 state があるとテスト間依存や並列実行の問題を起こしやすくなります。
- DB を使う integration test では、各テストでデータを独立させる、test ごとに transaction を戻す、database を初期化するなどの方針を決めます。

## 長すぎる Arrange の例

```csharp
[Fact]
public void CanOrder_ReturnsTrue_WhenStockIsEnough()
{
    var product = new Product(
        id: 1,
        name: "Keyboard",
        price: 12000m,
        stock: 10,
        isPublished: true,
        createdAt: DateTimeOffset.Parse("2026-05-22T00:00:00Z"));

    var result = product.CanOrder(3);

    Assert.True(result);
}
```

このテストで重要なのは `stock: 10` と `quantity: 3` です。他の値が多いと、仕様の焦点がぼやけます。

## test data builder の例

```csharp
public sealed class ProductBuilder
{
    private int id = 1;
    private string name = "Keyboard";
    private decimal price = 12000m;
    private int stock = 10;
    private bool isPublished = true;
    private DateTimeOffset createdAt = DateTimeOffset.Parse("2026-05-22T00:00:00Z");

    public ProductBuilder WithStock(int value)
    {
        stock = value;
        return this;
    }

    public ProductBuilder Unpublished()
    {
        isPublished = false;
        return this;
    }

    public Product Build()
    {
        return new Product(id, name, price, stock, isPublished, createdAt);
    }
}
```

## builder を使うテスト

```csharp
[Fact]
public void CanOrder_ReturnsFalse_WhenStockIsZero()
{
    var product = new ProductBuilder()
        .WithStock(0)
        .Build();

    var result = product.CanOrder(1);

    Assert.False(result);
}
```

## fixture を使う場面

```csharp
public sealed class DatabaseFixture : IAsyncLifetime
{
    public Task InitializeAsync()
    {
        // DB container の起動や migration をここで行います。
        return Task.CompletedTask;
    }

    public Task DisposeAsync()
    {
        // test resource の後片付けをここで行います。
        return Task.CompletedTask;
    }
}
```

## コードの読み方

`ProductBuilder` は、テストでよく使う既定値を持っています。テストでは `WithStock(0)` のように重要な値だけを変更しているため、「在庫が0の場合」を検証していることが読みやすくなります。fixture は DB や container のように初期化が重い resource を扱うための置き場所ですが、共有 state を持たせすぎないことが重要です。

## 実務での使い方

業務 object が複雑になるほど、test data builder が効きます。テストごとに重要な値だけを明示し、その他は安全な既定値にします。integration test では、データの重複、削除漏れ、並列実行の衝突を避けるため、ID や名前に test ごとの一意値を使うことがあります。

## よくあるミス

- 全テストで同じ mutable object を共有する。
- object mother に大量の意味を詰め込み、どの条件のデータか分からなくなる。
- test data builder の既定値が業務的に不正で、各テストが余計な設定を必要とする。
- integration test のデータ初期化と後片付けを決めない。
- テストデータが production の個人情報や秘密情報を含む。
