# テスト戦略

## 目的

Unit test、integration test、E2E test の役割を分け、何をどの粒度で検証するか判断できるようにします。

## 前提

- [xUnit の基本](01_xUnitの基本.md) を読んでいる
- [テストしやすい設計](04_テストしやすい設計.md) を読んでいる

## 要点

- Unit test は小さく速く、業務ルールの分岐を守ります。
- Integration test は DB、file、HTTP など境界との接続を確認します。
- E2E test は主要な user flow を少数だけ確認します。
- すべてを E2E に寄せると遅く不安定になり、すべてを unit test に寄せると接続事故を見逃します。

## コード例

```csharp
// この例では業務ルールだけを unit test として確認します。
[Fact]
public void CanOrder_ReturnsFalse_WhenStockIsZero()
{
    var product = new Product(stock: 0);

    var result = product.CanOrder(quantity: 1);

    Assert.False(result);
}
```

## コードの読み方

DB や HTTP を使わず、`Product` の業務判断だけを検証しています。このような test は速く、失敗したときの原因が狭いため、まず unit test に向いています。

## 判断表

| 検証したいこと | 向いている test | 例 |
| --- | --- | --- |
| 計算、分岐、validation | Unit test | 金額計算、在庫判定 |
| DB mapping、SQL、migration | Integration test | EF Core + Testcontainers |
| API contract | Integration test | `WebApplicationFactory` で HTTP 呼び出し |
| 画面を含む主要 flow | E2E test | ログインして注文する |

## 実務での使い方

pull request では unit test と主要 integration test を毎回動かし、重い E2E は nightly や release 前に寄せることがあります。flaky test は放置せず、時刻、乱数、外部サービス、並列実行の影響を取り除きます。

Web API の integration test は [Web API 統合テスト](../10_WebとAPI/21_WebAPI統合テスト.md)、flaky test の具体的な直し方は [flaky test 対策](17_flaky test対策.md) で扱います。

## よくあるミス

- DB が必要な処理を無理に mock しすぎる。
- すべてを E2E にして feedback が遅くなる。
- test data を各 test に直書きし、変更に弱くする。
- 並列実行で共有 DB や static state が衝突する。
