# CI でテストを実行する

## 目的

GitHub Actions で `dotnet test` を自動実行し、変更が壊れていないことを継続的に確認します。

## 前提

- [テストプロジェクト構成](08_テストプロジェクト構成.md) を読んでいる
- [GitHub Actions](../11_ツールと運用/05_GitHubActions.md) を読んでいる

## 要点

- CI は push や pull request のたびに同じ検証を実行します。
- `restore`、`build`、`test` を分けると失敗箇所を追いやすくなります。
- SDK バージョンはローカルと CI で揃えます。

## workflow 例

```yaml
# workflow の表示名です。
name: test

# push と pull request のたびに実行します。
on:
  push:
  pull_request:

jobs:
  test:
    # Microsoft-hosted runner の Ubuntu 環境で実行します。
    runs-on: ubuntu-latest
    steps:
      # repository のソースコードを runner に取得します。
      - uses: actions/checkout@v6

      # .NET SDK をインストールします。global.json と揃えると差分を減らせます。
      - uses: actions/setup-dotnet@v5
        with:
          dotnet-version: '10.0.x'

      # NuGet package を復元します。
      - run: dotnet restore

      # restore 済みなので --no-restore を付け、build の失敗だけを見やすくします。
      - run: dotnet build --no-restore --configuration Release

      # build 済みの成果物に対して test を実行し、coverage も収集します。
      - run: dotnet test --no-build --configuration Release --collect:"XPlat Code Coverage"
```

## workflow の読み方

この workflow は `restore`、`build`、`test` を分けています。失敗した step を見るだけで、依存関係の復元、コンパイル、テストのどこで壊れたかを切り分けやすくなります。

`dotnet-version` はローカルの SDK 方針と揃えます。`global.json` を使っている場合は、`global-json-file: ./global.json` を指定するか、`dotnet-version` と `global.json` の両方を同じ方針にします。両方を指定すると、それぞれの version がインストール対象になる点に注意します。

古い self-hosted runner や GitHub Enterprise Server では、最新 major の Action が使えない場合があります。その場合は runner の対応 version を確認してから `actions/setup-dotnet` の major version を決めます。

## 実務での使い方

PR 前にローカルで `dotnet test` を通し、PR では CI が同じ検証を実行します。失敗した場合は、ログから restore、build、test のどこで壊れたか確認します。

## よくあるミス

- CI だけ SDK バージョンが違う。
- `dotnet test --no-build` の前に build していない。
- 環境変数や secrets が未設定で integration test が落ちる。
- flaky test を rerun だけでごまかす。

## レビュー観点

- PR の必須チェックに test workflow が入っているか。
- CI ログに secret が出ていないか。
- integration test を実行する条件が明確か。
- coverage を収集する場合、結果の確認場所があるか。
