# カバレッジ測定

## 目的

テストがどのコードを実行したかを測定し、重要な仕様の抜けを見つける手がかりにします。

## 前提

- [xUnit の基本](01_xUnitの基本.md) を読んでいる
- [テストプロジェクト構成](08_テストプロジェクト構成.md) を読んでいる

## 要点

- カバレッジは品質そのものではなく、テスト不足を見つける指標です。
- 行カバレッジだけでなく、分岐や重要なエラーケースを確認します。
- 数字を上げるためだけのテストは保守コストになります。

## coverlet の例

```powershell
dotnet test --collect:"XPlat Code Coverage"
```

必要に応じてレポート生成ツールを追加します。

```powershell
dotnet tool install --global dotnet-reportgenerator-globaltool
reportgenerator -reports:**/coverage.cobertura.xml -targetdir:coverage-report -reporttypes:Html
```

## 見るポイント

- 業務ルールの正常系と異常系が通っているか。
- `if` や `switch` の重要な分岐が通っているか。
- 例外や validation のテストがあるか。
- ただ getter / setter を通して数字を稼いでいないか。

## 実務での使い方

PR で大きな変更が入ったとき、重要なロジックの未テスト部分を探すために使います。CI で coverage を出す場合も、閾値はチームの成熟度に合わせて段階的に設定します。

coverage を品質ゲートに入れる場合も、数字だけでなく [境界値と例外のテスト](14_境界値と例外のテスト.md) や [品質ゲートと PR チェック](18_品質ゲートとPRチェック.md) の観点と合わせて確認します。

## よくあるミス

- カバレッジ率だけで品質を判断する。
- 重要な業務分岐が未テストでも全体率が高いので見逃す。
- 自動生成コードや DTO まで無理に対象にする。
- flaky なテストを増やして数字だけ維持する。

## レビュー観点

- 変更された仕様に対応するテストがあるか。
- 境界値、null、空、例外ケースが含まれるか。
- coverage の低い箇所が、本当にリスクの低いコードか。
- CI で結果を確認できるか。

## 練習問題

1. `dotnet test --collect:"XPlat Code Coverage"` を実行する。
2. HTML レポートを生成する。
3. 未テストの分岐を1つ見つけてテストを追加する。
4. カバレッジ率ではなく、仕様の観点で足りないテストを説明する。
