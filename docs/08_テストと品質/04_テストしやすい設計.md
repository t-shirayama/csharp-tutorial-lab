# テストしやすい設計

## 目的

テストしやすいコードの特徴を理解し、実装時点で品質を上げられるようにします。

## 要点

- 入力と出力が明確な処理はテストしやすいです。
- 現在時刻、乱数、外部 API、DB は直接使わず差し替えます。
- 大きなメソッドや多すぎる依存はテスト困難のサインです。

## 実務での使い方

業務ロジックを Controller や DB アクセスから分けると、ユニットテストしやすくなります。副作用を境界に寄せるのが基本です。

solution と test project の分け方は [08_テストプロジェクト構成.md](08_テストプロジェクト構成.md)、DB を含むテストは [10_Testcontainers.md](10_Testcontainers.md) で扱います。

## よくあるミス

- static な現在時刻を直接呼ぶ。
- コンストラクタ内で外部接続する。
- ロジックと I/O が密結合している。

## テスト不可能になりやすいサイン

- メソッド内で `DateTime.Now` や `Guid.NewGuid()` を直接呼ぶ。
- メソッド内で `new HttpClient()` や DB 接続を作る。
- 戻り値がなく、内部状態や外部 I/O だけを変更する。
- private メソッドに重要な業務ルールが閉じ込められている。
- 1つの service が validation、DB、外部 API、ログ、変換をすべて担当している。

## レビュー観点

- 業務ルールが pure なメソッドまたは小さな service として切り出されているか。
- 現在時刻、乱数、外部 API、DB が差し替え可能か。
- テスト名で仕様が読めるか。
- エラーケース、境界値、null、空コレクションのテストがあるか。
