# テストプロジェクト構成

## 目的

アプリ本体とテストを分けた solution 構成を作り、`dotnet test` で検証できるようにします。

## 前提

- [xUnit の基本](01_xUnitの基本.md) を読んでいる
- [solution と project](../11_ツールと運用/03_solutionとproject.md) を読んでいる

## 要点

- 実務ではアプリ本体とテストプロジェクトを分けます。
- テストプロジェクトから本体プロジェクトを project reference します。
- solution に両方を追加すると、`dotnet test` でまとめて検証できます。

## 推奨構成

```text
TodoApp/
  TodoApp.sln
  src/
    TodoApp/
      TodoApp.csproj
  tests/
    TodoApp.Tests/
      TodoApp.Tests.csproj
```

## 作成コマンド

```powershell
dotnet new sln -n TodoApp
New-Item -ItemType Directory -Path src, tests

dotnet new classlib -n TodoApp -o src/TodoApp
dotnet new xunit -n TodoApp.Tests -o tests/TodoApp.Tests

dotnet sln add src/TodoApp/TodoApp.csproj
dotnet sln add tests/TodoApp.Tests/TodoApp.Tests.csproj
dotnet add tests/TodoApp.Tests/TodoApp.Tests.csproj reference src/TodoApp/TodoApp.csproj

dotnet test
```

## 実務での使い方

Web API でも、業務ロジックを class library に分けるとテストしやすくなります。小規模なら Web API project に直接 service を置いても構いませんが、テスト対象を Controller や endpoint から切り離す意識が重要です。

## よくあるミス

- テストプロジェクトを solution に追加し忘れる。
- app project が test project を参照してしまう。
- テスト用コードを本番 project に混ぜる。
- test project 名や namespace が本体と対応していない。

## レビュー観点

- `dotnet test` を solution root で実行できるか。
- 本体 project と test project の依存方向が一方向か。
- テストデータや helper が整理されているか。
- integration test と unit test の置き場所が分かるか。
