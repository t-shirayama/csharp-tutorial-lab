# 品質ゲートと PR チェック

## 目的

Pull Request 前後で確認する品質ゲートを整理し、テスト、format、静的解析、レビュー観点を実務の流れに組み込めるようにします。

## 前提

- [CI でテストを実行する](12_CIでテストを実行する.md) を読んでいる
- [dotnet format](05_dotnet-format.md) を読んでいる
- [静的解析](06_静的解析.md) を読んでいる
- [コードレビュー観点](07_コードレビュー観点.md) を読んでいる

## 要点

- 品質ゲートは、変更を main branch に入れる前に通す確認です。`dotnet restore`、`dotnet build`、`dotnet test`、format、analyzer、coverage、review を組み合わせます。
- ローカルで実行する確認と CI で実行する確認を揃えると、PR 後の手戻りを減らせます。
- `dotnet format --verify-no-changes` は、format 済みかどうかを CI で確認するのに使えます。自動修正を CI で勝手に commit するより、開発者が手元で直す運用が分かりやすいです。
- warning をすべて error にするかはチームの成熟度に合わせます。既存 warning が多い project では、まず新規 warning を増やさない方針から始めます。
- coverage は数字だけで判断しません。重要な業務分岐、例外、境界値がテストされているかを review します。
- PR checklist は、機械で検出しにくい仕様、運用、セキュリティ、ドキュメント更新を確認するために使います。

## ローカル確認コマンド

```powershell
dotnet restore
dotnet build --configuration Release
dotnet test --configuration Release
dotnet format --verify-no-changes
```

## CI workflow の例

```yaml
name: quality-gate

on:
  pull_request:
  push:
    branches: [ main ]

jobs:
  quality-gate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6

      - uses: actions/setup-dotnet@v5
        with:
          dotnet-version: '10.0.x'

      - run: dotnet restore

      - run: dotnet build --no-restore --configuration Release

      - run: dotnet test --no-build --configuration Release --collect:"XPlat Code Coverage"

      - run: dotnet format --verify-no-changes --verbosity minimal
```

## .editorconfig の例

```ini
root = true

[*.cs]
nullable = enable
dotnet_analyzer_diagnostic.category-Style.severity = warning
dotnet_diagnostic.CA2016.severity = warning
```

`CA2016` は `CancellationToken` の転送漏れを検出する analyzer rule です。全 rule を一気に厳しくするのではなく、効果の大きいものから段階的に導入します。

## PR checklist の例

```markdown
## 確認したこと

- [ ] `dotnet test` が成功した
- [ ] format / analyzer の指摘を確認した
- [ ] 変更した仕様に対応するテストを追加・更新した
- [ ] 境界値、null、空、例外ケースを確認した
- [ ] ログに秘密情報や個人情報が出ない
- [ ] 設定値、環境変数、secret の扱いを確認した
- [ ] API 変更がある場合は OpenAPI / docs を更新した
- [ ] DB migration がある場合は rollback / deploy 手順を確認した
```

## コードの読み方

workflow は `restore`、`build`、`test`、`format` を分けています。失敗した step を見れば、依存関係、コンパイル、テスト、整形のどこで問題が出たか分かります。PR checklist は、CI では検出しにくい仕様や運用リスクを人間が確認するためのものです。

## 実務での使い方

品質ゲートは厳しければよいわけではありません。最初は `build` と `test` を必須にし、次に format、analyzer、coverage、security scan を段階的に追加します。失敗が多すぎて誰も見なくなる gate は逆効果なので、チームが運用できる粒度にします。

## よくあるミス

- CI だけに頼り、ローカルで何も確認しない。
- format 差分と機能差分を同じ PR に混ぜる。
- analyzer warning を大量に放置し、重要な warning が埋もれる。
- coverage 閾値だけを満たし、重要な仕様のテストがない。
- PR checklist が形骸化し、実際には確認していない。

## 練習問題

1. PR 前に実行するローカル確認コマンドを README に書く。
2. `dotnet format --verify-no-changes` を CI に追加する。
3. 自分の project 用の PR checklist を作る。
4. coverage の数字ではなく、重要な分岐がテストされているか確認する。
