# デプロイ失敗時の診断

## 目的

GitHub Actions、Docker、アプリ起動、設定値、health check のどこで失敗しているかを切り分け、復旧までの手順を説明できるようにします。

## 前提

- [GitHub Actions](05_GitHubActions.md) を読んでいる
- [Docker](06_Docker.md) を読んでいる
- [デプロイ前チェックリスト](11_デプロイ前チェックリスト.md) を読んでいる

## 要点

- まず「build 失敗」「test 失敗」「image 作成失敗」「起動失敗」「health check 失敗」「runtime error」に分けます。
- GitHub Actions では失敗した step、exit code、直前の command、artifact を確認します。
- Docker では image tag、environment variables、port、entrypoint、log を確認します。
- 起動後に落ちる場合は、設定値、secret、DB 接続、migration、権限を疑います。
- rollback 手順と、再デプロイしてよい条件を事前に決めておきます。

## GitHub Actions で見る場所

```powershell
# local で同じ確認を先に再現します。
dotnet restore
dotnet build --configuration Release --no-restore
dotnet test --configuration Release --no-build
```

## Docker 起動を確認する

```powershell
docker build -t todo-api:local .
docker run --rm -p 8080:8080 --env ASPNETCORE_ENVIRONMENT=Production todo-api:local
docker logs <container-id>
```

## 設定値を確認する

```csharp
var connectionString = builder.Configuration.GetConnectionString("DefaultConnection");

if (string.IsNullOrWhiteSpace(connectionString))
{
    throw new InvalidOperationException("DefaultConnection is not configured.");
}
```

## コードの読み方

PowerShell の例は CI と同じ restore、build、test を local で再現するための最小手順です。Docker の例では image を作り、port と environment を指定して起動しています。C# の例では、起動時に必須設定が空なら早く失敗させます。設定漏れを後続処理の null reference として出すより、原因が明確になります。

## 実務での使い方

障害時は、ログを全部読むより先に失敗地点を分類します。build で落ちているなら code / package、起動で落ちているなら configuration / secret / DB、health check で落ちているなら network / port / dependency を疑います。復旧が優先なら rollback、原因調査が優先なら同じ artifact を staging で再現します。

## よくあるミス

- CI の失敗 step を見ずに関係ない箇所を修正する。
- local の Debug build だけ確認し、Release build を確認しない。
- secret が空のまま起動し、別の例外として表面化させる。
- container の port とアプリの listen port を混同する。
- rollback 手順がなく、修正 commit を重ねて復旧を遅らせる。
