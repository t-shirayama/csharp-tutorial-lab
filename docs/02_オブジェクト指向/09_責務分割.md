# 責務分割

## 目的

クラスやメソッドが担当する仕事を分け、変更しやすくテストしやすい形にします。

## 前提

- [class と object](01_classとobject.md) を読んでいる
- [SOLID 原則](../07_設計と実務パターン/01_SOLID原則.md) を読んでいる

## 要点

- 1つのクラスに複数の変更理由を持たせないようにします。
- 入力検証、業務判断、永続化、表示は分けて考えます。
- 責務名をクラス名やメソッド名で説明できない場合は分割を検討します。

## 分割例

```csharp
// この例では「責務分割」の要点を最小のコードで確認します。
public class TodoService
{
    private readonly ITodoRepository repository;

    public TodoService(ITodoRepository repository)
    {
        this.repository = repository;
    }

    public TodoItem Add(string title)
    {
        if (string.IsNullOrWhiteSpace(title))
        {
            throw new ArgumentException("Title is required.", nameof(title));
        }

        var todo = new TodoItem(title);
        repository.Save(todo);
        return todo;
    }
}
```

## コードの読み方

このコード例は「責務分割」の基本形を確認するためのものです。上から順に、値や object を用意し、C# の構文や .NET API を使い、最後に結果を確認します。まず入力、処理、出力の 3 つに分けて読むと、初学者でも流れを追いやすくなります。

## 実務での使い方

Controller や endpoint に業務ロジックが増え始めたら service へ移します。DB 操作が混ざるなら repository や DbContext 境界も意識します。

## よくあるミス

- `Manager` や `Helper` に何でも入れる。
- 分割しすぎて処理の流れが追えなくなる。
- 責務ではなくファイル行数だけで分割する。

## レビュー観点

- クラス名から責務が分かるか。
- 変更理由が複数混ざっていないか。
- テストしたい業務ルールが I/O から分離されているか。

## 練習問題

1. 入力、計算、保存が混ざったメソッドを3つに分ける。
2. `TodoService` と `TodoRepository` の責務を書く。
3. `Manager` という名前のクラスを具体名へ変更する。
