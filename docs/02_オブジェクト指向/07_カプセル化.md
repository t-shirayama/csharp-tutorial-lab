# カプセル化

## 目的

オブジェクトの状態を守り、不正な変更を防ぐためのカプセル化を理解します。

## 要点

- カプセル化は、状態と状態を変更する処理をまとめる考え方です。
- 外から自由に書き換えられるプロパティは、不整合の原因になります。
- `private set`、読み取り専用プロパティ、意味のあるメソッドで状態を守ります。

## コード例

```csharp
// この例では「カプセル化」の要点を最小のコードで確認します。
var account = new BankAccount(1000m);
account.Deposit(500m);
account.Withdraw(300m);

Console.WriteLine(account.Balance);

public class BankAccount
{
    public BankAccount(decimal initialBalance)
    {
        if (initialBalance < 0) throw new ArgumentOutOfRangeException(nameof(initialBalance));
        Balance = initialBalance;
    }

    public decimal Balance { get; private set; }

    public void Deposit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentOutOfRangeException(nameof(amount));
        Balance += amount;
    }

    public void Withdraw(decimal amount)
    {
        if (amount <= 0 || amount > Balance) throw new InvalidOperationException("出金できません。");
        Balance -= amount;
    }
}
```

## コードの読み方

このコード例は「カプセル化」の基本形を確認するためのものです。上から順に、値や object を用意し、C# の構文や .NET API を使い、最後に結果を確認します。まず入力、処理、出力の 3 つに分けて読むと、初学者でも流れを追いやすくなります。

## 実務での使い方

注文、在庫、請求、支払いなど、状態の整合性が重要な処理で特に大切です。状態変更は `SetXxx` ではなく、業務上の意味を持つ `Cancel`, `Approve`, `AddItem` のようなメソッドにします。

## よくあるミス

- すべてのプロパティを `public set` にする。
- 状態変更のルールを呼び出し側に散らばらせる。
- private にすべき補助メソッドを public にする。

## 練習問題

1. `Stock` クラスを作り、在庫数を直接 set できないようにする。
2. 入庫と出庫のメソッドで在庫数を変更する。
3. 出庫数が在庫数を超えた場合の扱いを決める。
