# PR 提出チェックリスト

## 目的

実装後に、レビューされやすく安全な PR として提出する観点を整理します。

## 作るもの

自分の変更を説明する PR 本文と、確認済み項目のチェックリストです。

## 学習ポイント

- 変更目的を説明する
- 確認コマンドを明記する
- 仕様変更、破壊的変更、未対応事項を隠さない
- スクリーンショットや request/response 例を添える

## チェックリスト

- 何を、なぜ変更したかを書いた。
- `dotnet build` を実行した。
- `dotnet test` を実行した。
- 追加・変更した仕様にテストがある。
- 設定値、secret、migration の影響を確認した。
- API 変更がある場合、OpenAPI と README を更新した。
- ログに個人情報や secret が出ないことを確認した。

## PR 本文例

```markdown
## Summary

- TODO 作成 API を追加
- 入力 validation と ProblemDetails response を追加

## Checks

- dotnet build
- dotnet test

## Notes

- DB migration あり
```

## 完了条件

- レビュー担当者が変更内容と確認方法をすぐ把握できる。
- CI 結果とローカル確認が一致している。
- リスクや残課題が明記されている。

## 発展課題

- PR template を作る。
- review 指摘を分類して再発防止メモにする。
- release note を PR から作る。
