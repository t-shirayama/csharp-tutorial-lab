# 初期トラブルシュート

## 目的

.NET の環境構築直後に起きやすい問題を、コマンドで切り分けられるようにします。

## 前提

- Windows と PowerShell を使う
- VS Code で C# を編集する

## 要点

- まず terminal で `dotnet --info` を確認します。
- VS Code の問題か、PowerShell の問題か、SDK の問題かを分けます。
- PATH 更新後は VS Code の再起動が必要な場合があります。
- エラー文は省略せず記録します。

## 最初に確認するインストール状態

| 対象 | 確認方法 | 正常な状態 |
| --- | --- | --- |
| .NET SDK | `dotnet --list-sdks` | `10.0.xxx` など SDK が表示される |
| dotnet の場所 | `Get-Command dotnet` | `C:\Program Files\dotnet\dotnet.exe` などが表示される |
| VS Code | `code --version` | VS Code の version が表示される |
| C# Dev Kit | VS Code 拡張機能画面 | C# Dev Kit が Enabled |

どれか1つでも確認できない場合は、[06_インストール手順チェックリスト.md](06_インストール手順チェックリスト.md) に戻って順番に入れ直します。

## 確認コマンド

```powershell
dotnet --info
dotnet --list-sdks
Get-Command dotnet
$env:Path -split ';'
```

## 切り分け表

| 症状 | 確認 | 次の対応 |
| --- | --- | --- |
| `dotnet` が見つからない | `Get-Command dotnet` | PATH と SDK インストールを確認 |
| VS Code だけ認識しない | 外部 PowerShell で `dotnet --info` | VS Code を再起動 |
| build が失敗する | 最初の error を読む | コード、SDK、NuGet を分ける |
| 補完が効かない | C# Dev Kit と solution | 拡張機能と project 読み込みを確認 |
| `code` が見つからない | `code --version` | VS Code の PATH 設定を確認 |

## 入れ直す場合の順序

1. VS Code と PowerShell をすべて閉じる。
2. .NET SDK x64 をインストールする。
3. VS Code User Installer x64 をインストールする。
4. VS Code を開き、C# Dev Kit をインストールする。
5. VS Code と PowerShell を開き直す。
6. `dotnet --info`、`code --version`、`dotnet new console` を確認する。

## 実務での使い方

環境差分は再現性が重要です。発生した端末、実行したディレクトリ、コマンド、エラー全文、`dotnet --info` を残すと、チームで調査しやすくなります。

## よくあるミス

- エラーの最後の行だけ見て判断する。
- VS Code を再起動せず PATH 反映を疑い続ける。
- SDK ではなく runtime だけ入れている。
- project を開かず、単体ファイルだけ開いて補完を期待する。
- 公式以外の古い installer や記事に従って、不要な runtime だけを入れる。

## 練習問題

1. `dotnet --info` の SDK と runtime の違いを確認する。
2. `Get-Command dotnet` の結果を読む。
3. VS Code と外部 PowerShell で同じコマンドを実行する。
4. トラブル報告に必要な情報を箇条書きにする。
