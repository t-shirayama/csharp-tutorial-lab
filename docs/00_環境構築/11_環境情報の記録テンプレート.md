# 環境情報の記録テンプレート

## 目的

環境構築や build / restore / debug の問題を相談するときに、必要な情報を漏れなく記録できるようにします。

## 前提

- [初期トラブルシュート](05_初期トラブルシュート.md) を読んでいる
- PowerShell で確認コマンドを実行できる

## 要点

- 環境問題は、OS、SDK、VS Code、repository、実行 directory、実行 command、エラー全文がそろうと切り分けやすくなります。
- 「動きません」だけでは原因を絞れません。期待した結果、実際の結果、再現手順を分けて書きます。
- エラーは省略せず、最初の error と stack trace を含めて記録します。ただし secret、token、password、社内 URL は共有前に伏せます。
- VS Code の統合 terminal と外部 PowerShell で結果が違う場合は、PATH 反映や起動タイミングの問題を疑います。
- `dotnet --info`、`dotnet --list-sdks`、`Get-Command dotnet`、`git status` は、初期調査でよく使います。
- 相談時は、直前に変更したことも書きます。SDK 更新、extension 更新、branch 変更、package 追加などが手がかりになります。

## 確認コマンド

```powershell
dotnet --info
dotnet --list-sdks
Get-Command dotnet
code --version
git status
```

NuGet restore の問題では、package source も確認します。

```powershell
dotnet nuget list source
```

## 記録テンプレート

```text
## 何をしたいか
例: repository を clone して dotnet build を通したい。

## 実行環境
OS:
PowerShell version:
VS Code version:
C# Dev Kit version:
.NET SDK:
Repository / branch:
実行 directory:

## 実行したコマンド
例:
dotnet restore
dotnet build

## 期待した結果
例: build が成功する。

## 実際の結果
例: NU1301 が出て restore に失敗する。

## エラー全文
ここに省略せず貼る。secret、token、password は伏せる。

## 直前に変更したこと
例: SDK を更新した、branch を切り替えた、NuGet source を追加した。

## 試したこと
例: VS Code 再起動、外部 PowerShell で dotnet --info 確認。
```

## dotnet --info の見方

`dotnet --info` では、主に次を確認します。

| 項目 | 見る理由 |
| --- | --- |
| SDK Version | build に使われる SDK の候補を確認する |
| Base Path | `dotnet` が参照している SDK の場所を確認する |
| Host Version | runtime 側の version を確認する |
| OS Name / Version | Windows version 差分を確認する |
| .NET SDKs installed | `global.json` が要求する SDK が入っているか確認する |
| .NET runtimes installed | 実行に必要な runtime があるか確認する |

## コードの読み方

テンプレートは、再現に必要な情報を上から順に並べています。`実行したコマンド` と `実行 directory` が分かると、同じ場所で同じ操作を再現できます。`期待した結果` と `実際の結果` を分けると、仕様の誤解なのか環境問題なのかを切り分けやすくなります。

## 実務での使い方

チームに質問するとき、issue を作るとき、障害調査を引き継ぐときに使います。自分のメモとして残しておくと、同じ問題が再発したときに短時間で復旧できます。環境構築直後は、成功した `dotnet --info` と `dotnet test` の結果も残しておくと比較しやすくなります。

## よくあるミス

- エラーの最後の1行だけを貼る。
- 実行 directory を書かない。
- VS Code terminal なのか外部 PowerShell なのかを書かない。
- secret や token を伏せずに共有する。
- 直前に変更した SDK、branch、package、extension を書き忘れる。

## 練習問題

1. 自分の環境で `dotnet --info` を実行し、SDK version を記録する。
2. `dotnet build` が失敗した想定で、テンプレートを埋める。
3. secret が含まれていないか確認する。
4. VS Code terminal と外部 PowerShell で `dotnet --info` の結果を比べる。
