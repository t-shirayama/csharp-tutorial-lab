# スレッド安全性

## 目的

複数の処理が同じ状態を同時に扱うときの危険と、基本的な回避策を理解します。

## 前提

- [Task.WhenAll](04_TaskWhenAll.md) を読んでいる
- [Parallel 処理](05_Parallel処理.md) を読んでいる

## 要点

- 共有 mutable state は競合の原因になります。
- `List<T>` や `Dictionary<TKey,TValue>` は同時書き込みに注意が必要です。
- 共有しない、immutable にする、lock する、thread-safe collection を使う、の順に検討します。

## 悪い例

```csharp
// List<T> は複数 task から同時に Add される前提の型ではありません。
var results = new List<int>();

await Task.WhenAll(items.Select(async item =>
{
    var value = await LoadAsync(item);

    // 同時書き込みが起きると、データ破損や例外の原因になります。
    results.Add(value);
}));
```

## 改善例

```csharp
// 各 task の戻り値を Task.WhenAll がまとめるため、共有 List へ書き込みません。
var results = await Task.WhenAll(items.Select(item => LoadAsync(item)));
```

## コードの読み方

悪い例では複数の task が同じ `results` に同時に書き込んでいます。改善例では、各 task が値を返し、`Task.WhenAll` が結果を配列としてまとめます。共有状態をなくすことで、lock なしでも安全に書けます。

## 実務での使い方

並列化するときは、先に正しい逐次処理を書きます。その後、共有状態を減らした形で並列化します。性能改善のために correctness を犠牲にしないことが重要です。

## よくあるミス

- 共有 `List<T>` に並列で `Add` する。
- `lock` すれば何でも安全だと思う。
- Singleton service に mutable state を持たせる。
- テストで再現しにくい race condition を見逃す。
