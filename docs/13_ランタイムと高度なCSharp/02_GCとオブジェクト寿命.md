# GC とオブジェクト寿命

## 目的

Garbage Collection と object lifetime の入口を理解します。

## 前提

- [IDisposable と using](../04_標準ライブラリ/11_IDisposableとusing.md) を読んでいる

## 要点

- GC は到達不能になった managed object のメモリを回収します。
- object が参照され続ける限り、GC の対象になりません。
- `GC.Collect()` を通常の業務コードで明示的に呼ぶことはほとんどありません。
- unmanaged resource は `IDisposable` で早めに解放します。

## コード例

```csharp
// この例では「GC とオブジェクト寿命」の要点を最小のコードで確認します。
var before = GC.GetTotalMemory(forceFullCollection: false);

var values = Enumerable.Range(0, 100_000)
    .Select(number => number.ToString())
    .ToArray();

var after = GC.GetTotalMemory(forceFullCollection: false);

Console.WriteLine($"Before: {before}");
Console.WriteLine($"After : {after}");
```

## コードの読み方

大量の文字列を作ると managed heap の使用量が増えます。GC は不要になった object を回収できますが、参照が残っている `values` はまだ到達可能です。

## 実務での使い方

メモリリーク調査、large object heap、短命 object の大量生成、cache の設計、event subscription の解除漏れで関係します。まずは object を不要に長く参照しない設計を意識します。

## よくあるミス

- GC があるから `Dispose` は不要だと思う。
- static collection に object を溜め続ける。
- event 購読を解除せず、参照が残り続ける。
- 原因調査なしに `GC.Collect()` を入れる。

## 練習問題

1. 大量の object を作って `GC.GetTotalMemory` を確認する。
2. static list に保持した場合と保持しない場合を比較する。
3. `IDisposable` が必要な resource と GC が扱う memory の違いを説明する。
