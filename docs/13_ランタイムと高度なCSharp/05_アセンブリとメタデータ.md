# アセンブリとメタデータ

## 目的

.NET assembly、metadata、resource の入口を理解します。

## 前提

- [solution と project](../11_ツールと運用/03_solutionとproject.md) を読んでいる
- [reflection](../06_型システムと言語機能/06_reflection.md) を読んでいる

## 要点

- assembly は .NET の配置単位です。
- assembly には IL、metadata、resource などが含まれます。
- metadata により、reflection で型や member を調べられます。
- package、project、assembly は似ていますが同じではありません。

## コード例

```csharp
// この例では「アセンブリとメタデータ」の要点を最小のコードで確認します。
var assembly = typeof(string).Assembly;

Console.WriteLine(assembly.FullName);
Console.WriteLine(assembly.Location);
```

## コードの読み方

`typeof(string).Assembly` で、`string` 型が定義されている assembly を取得します。`FullName` には assembly 名、version、culture、public key token などが含まれます。

## 実務での使い方

plugin、reflection、依存関係調査、NuGet package の中身確認、version conflict 調査で出てきます。通常のアプリ開発では project / package として意識することが多いですが、実行時問題では assembly 単位の理解が必要です。

## よくあるミス

- NuGet package 1 つが assembly 1 つだと思い込む。
- project 名と assembly 名が常に一致すると思う。
- version conflict を package だけ見て判断し、実際に load された assembly を確認しない。
