# ファイナライゼーション

## 目的

finalizer の役割と、`IDisposable` との違いを理解します。

## 前提

- [IDisposable と using](../04_標準ライブラリ/11_IDisposableとusing.md) を読んでいる
- [GC とオブジェクト寿命](02_GCとオブジェクト寿命.md) を読んでいる

## 要点

- finalizer は GC が object を回収する前に呼び出す可能性がある後始末です。
- 呼び出しタイミングは予測できません。
- 通常の resource 解放は `IDisposable` と `using` を優先します。
- finalizer は unmanaged resource を直接持つ型など、限定的な場面で使います。

## コード例

```csharp
// この例では「ファイナライゼーション」の要点を最小のコードで確認します。
public sealed class NativeResourceHolder : IDisposable
{
    private bool disposed;

    ~NativeResourceHolder()
    {
        Dispose(disposing: false);
    }

    public void Dispose()
    {
        Dispose(disposing: true);
        GC.SuppressFinalize(this);
    }

    private void Dispose(bool disposing)
    {
        if (disposed)
        {
            return;
        }

        disposed = true;
    }
}
```

## コードの読み方

`Dispose` が明示的に呼ばれた場合は `GC.SuppressFinalize` で finalizer を不要にします。finalizer は保険に近く、通常の解放タイミングとして期待してはいけません。

## 実務での使い方

多くのアプリでは finalizer を自作しません。ファイル、DB、HTTP などは既存型が `IDisposable` を実装しているため、`using` で破棄します。unmanaged handle を直接包むライブラリを書くときに検討します。

## よくあるミス

- finalizer に通常の業務処理を書く。
- finalizer の実行タイミングを期待する。
- `Dispose` と finalizer の両方で二重解放する。
