# 例外処理の入口

## 目的

C# でエラーが起きたときの基本的な考え方と、`try`、`catch`、`finally`、`throw` の入口を理解します。

## 前提

- [条件分岐](03_条件分岐.md) を読んでいる
- [メソッド](05_メソッド.md) を読んでいる

## 要点

- 例外は、通常の処理を続けられない問題を表す仕組みです。ファイルが存在しない、外部 API が失敗した、DB 接続が切れたなど、呼び出し元へ異常を伝えるために使います。
- `try` で例外が起きる可能性のある処理を囲み、`catch` で扱います。`catch` では復旧できる処理、利用者へ返す message、ログ出力などを行い、何もできない例外は無理に捕まえません。
- 予測できる入力ミスは、例外よりも事前チェックや `TryParse` で扱うことが多いです。ユーザー入力の形式ミスはよく起きる通常ケースなので、例外に頼るとコードの意図が分かりにくくなります。
- 例外を握りつぶすと、原因調査が難しくなります。`catch { }` のように何もせず消すと、失敗した事実も stack trace も失われ、後続処理で別の不具合として表面化します。
- `throw;` と `throw ex;` は意味が違います。再スローする場合は `throw;` を使うと元の stack trace を保てるため、調査しやすくなります。
- 実務では、どの層で例外を捕まえるかを決めます。低い層では詳細を残して上位へ伝え、Web API の入口ではログを出して適切な HTTP status に変換する、という分担がよく使われます。

## 例外が起きる例

```csharp
// この例では「例外処理の入口」の要点を最小のコードで確認します。
var text = "abc";
var number = int.Parse(text);

Console.WriteLine(number);
```

`"abc"` は整数に変換できないため、`FormatException` が発生します。

## try / catch

```csharp
// この例では「例外処理の入口」の要点を最小のコードで確認します。
try
{
    var text = "abc";
    var number = int.Parse(text);

    Console.WriteLine(number);
}
catch (FormatException ex)
{
    Console.WriteLine($"数値に変換できません: {ex.Message}");
}
```

`catch` では、どの種類の例外を扱うかを明示できます。

## TryParse を使う

入力値の変換では、例外を待つより `TryParse` で判定する方が自然です。

```csharp
// この例では「例外処理の入口」の要点を最小のコードで確認します。
Console.Write("数値を入力してください: ");
var text = Console.ReadLine();

if (int.TryParse(text, out var number))
{
    Console.WriteLine($"入力値: {number}");
}
else
{
    Console.WriteLine("数値として解釈できません。");
}
```

## finally

`finally` は、例外が起きても起きなくても最後に実行されます。

```csharp
// この例では「例外処理の入口」の要点を最小のコードで確認します。
try
{
    Console.WriteLine("処理を開始します。");
}
finally
{
    Console.WriteLine("後片付けをします。");
}
```

ファイル、ネットワーク、DB 接続などの後片付けで関係します。ただし実務では `using` や `await using` を使う場面も多いです。

## throw

不正な状態を検出したとき、自分で例外を投げることもできます。

```csharp
// この例では「例外処理の入口」の要点を最小のコードで確認します。
static decimal CalculatePrice(decimal unitPrice, int quantity)
{
    if (unitPrice < 0)
    {
        throw new ArgumentOutOfRangeException(nameof(unitPrice), "単価は0以上にしてください。");
    }

    if (quantity < 0)
    {
        throw new ArgumentOutOfRangeException(nameof(quantity), "数量は0以上にしてください。");
    }

    return unitPrice * quantity;
}
```

## よくあるミス

- `catch (Exception)` で何でも捕まえて、そのまま何もしない。
- エラーを画面に出すだけで、呼び出し元に失敗を伝えない。
- 入力ミスのような通常あり得るケースまで例外に頼る。
- 例外メッセージだけを見て、例外の型やスタックトレースを確認しない。

## コードの読み方

`try / catch` の例では、`int.Parse(text)` が失敗する可能性のある処理です。`catch (FormatException ex)` は、数値形式に変換できない失敗だけを扱います。一方、`TryParse` の例では失敗を例外ではなく `false` として受け取り、通常の条件分岐で message を出しています。ユーザー入力ではこちらの方が自然です。

## 実務での使い方

実務では、例外はログ、API のエラーレスポンス、リトライ、トランザクションのロールバックと関係します。基礎段階では、まず「予測できる入力ミスは検証で扱う」「想定外の失敗は例外として記録する」という切り分けを意識します。

ファイルや stream の後片付けは、`try / finally` を手書きするより [using とリソース解放](17_usingとリソース解放.md) を使う場面が多いです。
