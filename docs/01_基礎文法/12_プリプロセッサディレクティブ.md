# プリプロセッサディレクティブ

## 目的

`#nullable`、`#if`、`#pragma` などのプリプロセッサディレクティブを読み、使いどころを判断できるようにします。

## 前提

- [変数と型](02_変数と型.md) を読んでいる
- [nullable reference types](../06_型システムと言語機能/01_nullable-reference-types.md) を読んでいる

## 要点

- プリプロセッサディレクティブは `#` で始まり、コンパイル時の扱いを変えます。
- `#if DEBUG` は条件付きコンパイルに使います。
- `#nullable` は nullable reference types の有効化や無効化に使います。
- `#pragma warning disable` は警告抑制に使えますが、理由を残すべきです。

## コード例

```csharp
// この例では「プリプロセッサディレクティブ」の要点を最小のコードで確認します。
#nullable enable

string? name = null;

#if DEBUG
Console.WriteLine("Debug build でだけ出力します。");
#endif

#pragma warning disable CS0168
try
{
    throw new InvalidOperationException();
}
catch (Exception ex)
{
    // 学習用の例です。実務では例外を握りつぶさないようにします。
}
#pragma warning restore CS0168
```

## コードの読み方

`#nullable enable` は null 安全性の解析を有効にします。`#if DEBUG` の中は DEBUG symbol が定義されている build でだけコンパイルされます。`#pragma warning disable` は警告を抑制しますが、乱用すると問題を隠します。

## 実務での使い方

nullable 移行、debug 用ログ、platform 差分、生成コードの警告抑制で見かけます。通常の業務ロジックの分岐には `if` を使い、コンパイル対象そのものを変える必要があるときだけ `#if` を検討します。

## よくあるミス

- `#pragma` で警告を消した理由を残さない。
- `#if` を多用し、どの build で何が動くか分からなくなる。
- `#region` で長すぎる class を隠してしまう。

## 練習問題

1. `#nullable enable` を入れたときに警告が出るコードを書く。
2. `#if DEBUG` で debug 時だけ出る message を追加する。
3. `#pragma` を使うなら、なぜ必要かコメントで説明する。
