# レイヤードアーキテクチャ

## 目的

責務ごとに層を分け、依存関係を整理する基本を理解します。

## 要点

- Presentation、Application、Domain、Infrastructure のように責務を分けます。Presentation は HTTP や UI、Application は use case、Domain は業務ルール、Infrastructure は DB や外部 API を担当します。
- 上位層が下位層の詳細に依存しすぎないようにします。特に Domain が `DbContext`、`HttpClient`、環境変数などを直接使うと、業務ルールのテストが難しくなります。
- 依存方向を意識するとテストしやすくなります。Application は repository interface に依存し、実装は Infrastructure に置くと、テストでは fake repository に差し替えられます。
- 小規模アプリでは層を増やしすぎないことも重要です。処理が単純なうちは薄い構成にし、業務ルールや外部依存が増えたら分ける判断をします。

## 層ごとの責務例

| 層 | 置くもの | 置かないもの |
| --- | --- | --- |
| Presentation | Controller、endpoint、Request / Response DTO | 業務ルール、SQL |
| Application | UseCase、Application Service、transaction 境界 | HTTP response の詳細 |
| Domain | Entity、Value Object、業務ルール | DbContext、HttpClient |
| Infrastructure | Repository 実装、外部 API client、file access | 業務判断 |

## 依存方向の例

```csharp
public class CompleteOrderService
{
 private readonly IOrderRepository orderRepository;

 public CompleteOrderService(IOrderRepository orderRepository)
 {
  this.orderRepository = orderRepository;
 }

 public async Task CompleteAsync(int orderId, CancellationToken cancellationToken)
 {
  var order = await orderRepository.FindByIdAsync(orderId, cancellationToken)
   ?? throw new InvalidOperationException("注文が見つかりません。");

  order.Complete();
  await orderRepository.SaveAsync(order, cancellationToken);
 }
}

public interface IOrderRepository
{
 Task<Order?> FindByIdAsync(int orderId, CancellationToken cancellationToken);
 Task SaveAsync(Order order, CancellationToken cancellationToken);
}
```

## コードの読み方

`CompleteOrderService` は `IOrderRepository` interface に依存しています。DB の具体的な保存方法は見えていません。これにより、Application 層は use case の流れに集中でき、Infrastructure 層の実装を差し替えやすくなります。

## 実務での使い方

Web API、業務ロジック、DB アクセス、外部連携を分けます。小規模アプリでは過剰分割に注意し、変更頻度と複雑さに応じて層を増やします。

## よくあるミス

- Controller に業務ロジックが集中する。
- Repository に業務判断が入り込む。
- 層を分けたのに依存が循環する。
