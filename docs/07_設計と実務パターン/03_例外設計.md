# 例外設計

## 目的

例外をどこで投げ、どこで捕まえ、どう利用者へ伝えるかを設計できるようにします。

## 要点

- 予測可能な入力ミスは validation として扱います。必須項目不足、形式不正、範囲外など、利用者が直せるものは 400 response や validation result にします。
- 業務ルール違反は domain error として扱います。在庫不足、状態遷移不可、重複登録などは、system error と区別できる形にします。
- 想定外や処理継続不能な状態は例外にします。DB 接続失敗、外部サービス障害、programming error などは、低レイヤーで握りつぶさず境界まで伝えます。
- 低レイヤーで握りつぶさず、アプリ境界でログやレスポンスに変換します。API では middleware や filter で ProblemDetails へ変換すると形式を統一しやすくなります。
- 例外 message は内部調査向けであり、外部利用者へそのまま返すものではありません。接続文字列、token、個人情報が含まれないようにします。

## コード例

```csharp
public enum CompleteOrderResult
{
    Completed,
    NotFound,
    AlreadyCanceled
}

public class Order
{
    public OrderStatus Status { get; private set; }

    public CompleteOrderResult Complete()
    {
        if (Status == OrderStatus.Canceled)
        {
            // 想定できる業務ルール違反は戻り値として表します。
            return CompleteOrderResult.AlreadyCanceled;
        }

        Status = OrderStatus.Completed;
        return CompleteOrderResult.Completed;
    }
}

public enum OrderStatus
{
    Draft,
    Completed,
    Canceled
}
```

```csharp
try
{
    await externalApiClient.SendAsync(cancellationToken);
}
catch (HttpRequestException exception)
{
    // 外部サービス障害は呼び出し元へ例外として伝え、境界でログとレスポンスに変換します。
    throw new ExternalServiceException("外部 API 呼び出しに失敗しました。", exception);
}

public class ExternalServiceException : Exception
{
    public ExternalServiceException(string message, Exception innerException)
        : base(message, innerException)
    {
    }
}
```

## コードの読み方

最初の例では、キャンセル済み注文を完了できないことを業務上の結果として返しています。2つ目の例では、外部 API 障害を独自例外に包み、上位層で扱いやすくしています。利用者が直せる失敗と、システム側で調査すべき失敗を分けるのがポイントです。

## 実務での使い方

Web API では例外を middleware や filter で捕まえ、適切な HTTP ステータスに変換します。業務エラーとシステムエラーを混ぜないことが重要です。

ASP.NET Core で一元的に例外を HTTP レスポンスへ変換する実装は [12_グローバルエラーハンドリング.md](12_グローバルエラーハンドリング.md) で扱います。

## よくあるミス

- `catch (Exception)` で何もしない。
- 例外メッセージに秘密情報を含める。
- どこでもログを出して二重三重に記録する。

## レビュー観点

- 入力エラー、業務エラー、システムエラー、外部サービス一時失敗を分けているか。
- 例外を捕まえる場所がアプリ境界に寄っているか。
- ログに個人情報、トークン、接続文字列が出ないか。
- API 利用者が修正できるエラーは ProblemDetails などで説明されているか。

## 練習問題

1. 入力エラーとシステムエラーを分類する。
2. 例外を投げる場所と捕まえる場所を決める。
3. API のエラーレスポンス方針を書く。
