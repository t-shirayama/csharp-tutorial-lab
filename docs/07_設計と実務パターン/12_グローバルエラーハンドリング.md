# グローバルエラーハンドリング

## 目的

ASP.NET Core で例外処理を一元化し、ログと HTTP エラーレスポンスを統一できるようにします。

## 前提

- [例外設計](03_例外設計.md) を読んでいる
- [ILogger と構造化ログ](11_ILoggerと構造化ログ.md) を読んでいる

## 要点

- 例外は各 endpoint でばらばらに捕まえず、アプリ境界でまとめて扱います。
- API では ProblemDetails でエラーレスポンス形式を統一します。
- 予測可能な入力エラーは validation、想定外の失敗は例外として扱います。
- ログには調査に必要な ID を入れ、秘密情報は出しません。

## 最小例

```csharp
// この例では「グローバルエラーハンドリング」の要点を最小のコードで確認します。
var builder = WebApplication.CreateBuilder(args);

// ProblemDetails を DI に登録し、エラーレスポンスの標準形式を使えるようにします。
builder.Services.AddProblemDetails();

var app = builder.Build();

// 以降の pipeline で発生した未処理例外をまとめて扱います。
app.UseExceptionHandler();

app.MapGet("/error-sample", () =>
{
    // 動作確認用に、意図的に未処理例外を発生させます。
    throw new InvalidOperationException("Sample failure.");
});

app.Run();
```

ASP.NET Core の機能で ProblemDetails を返す入口です。業務エラーの分類やログ方針は別途設計します。

## コードの読み方

`AddProblemDetails` は ProblemDetails を生成するためのサービス登録です。`UseExceptionHandler` は request pipeline に例外処理を組み込みます。

この最小例は「例外を endpoint ごとに捕まえない」形を示しています。実務では、業務例外、validation、想定外例外を分類し、HTTP status とログ方針を揃えます。

## カスタム例外の例

```csharp
// この例では「グローバルエラーハンドリング」の要点を最小のコードで確認します。
public class DomainException : Exception
{
    // 業務ルール違反を表す message を基底 Exception に渡します。
    public DomainException(string message) : base(message)
    {
    }
}
```

業務ルール違反を表す例外は、システム例外と区別できる名前にします。

## エラー分類の目安

| 分類 | 例 | HTTP の目安 |
| --- | --- | --- |
| validation error | 必須項目不足、形式不正 | 400 |
| domain error | 在庫不足、状態遷移不可 | 400 または 409 |
| authentication error | 未ログイン | 401 |
| authorization error | 権限不足 | 403 |
| not found | 対象なし | 404 |
| system error | 想定外例外 | 500 |
| transient error | 外部サービス一時失敗 | 503 |

## 実務での使い方

Web API のエラーレスポンスは利用者との契約です。endpoint ごとに文字列を返すのではなく、形式、ステータス、エラーコード、trace id を統一します。

## よくあるミス

- endpoint ごとに `try/catch` を書く。
- すべての例外を 500 にする。
- 例外メッセージをそのまま外部公開する。
- API 利用者が修正できる情報を返さない。

## レビュー観点

- エラー分類と HTTP ステータスが対応しているか。
- API レスポンスに trace id や error code があるか。
- ログとレスポンスで情報量を分けているか。
- validation、domain、system error が混ざっていないか。
