# アプリケーションサービスと責務分割

## 目的

Controller、Application Service、Domain、Repository の責務を分け、業務処理をどこへ書くか判断できるようにします。

## 前提

- [DI の考え方](02_DIの考え方.md) を読んでいる
- [レイヤードアーキテクチャ](07_レイヤードアーキテクチャ.md) を読んでいる
- [例外設計](03_例外設計.md) を読んでいる

## 要点

- Controller や endpoint は HTTP の入口です。request を受け取り、認証情報や route parameter を取り出し、Application Service を呼び、response に変換する責務に寄せます。
- Application Service は use case の流れを表します。入力検証、domain object の取得、業務処理の呼び出し、永続化、transaction 境界などを調整します。
- Domain は業務ルールを表します。注文を完了できる条件、在庫を引き当てる条件、状態遷移の可否など、HTTP や DB に依存しない判断を置きます。
- Repository は永続化の入口です。DB の詳細を隠し、Application Service からは `GetByIdAsync` や `SaveAsync` のような業務寄りの操作として使います。
- 責務分割の目的は class を増やすことではありません。変更理由を分け、テストしやすくし、Controller や DB access に業務判断が散らばるのを防ぐことです。
- 小さな CRUD では過剰分割に注意します。業務ルールが薄い処理では、Application Service を薄く保ち、複雑になった use case から分割する方が現実的です。

## 悪い例

```csharp
app.MapPost("/orders/{id:int}/complete", async (int id, AppDbContext dbContext) =>
{
    // HTTP の入口に、DB access と業務判断が直接書かれています。
    var order = await dbContext.Orders.FindAsync(id);
    if (order is null)
    {
        return Results.NotFound();
    }

    if (order.Status == "Canceled")
    {
        return Results.BadRequest("キャンセル済みの注文は完了できません。");
    }

    order.Status = "Completed";
    order.CompletedAt = DateTimeOffset.UtcNow;

    await dbContext.SaveChangesAsync();
    return Results.NoContent();
});
```

## 改善例

```csharp
app.MapPost("/orders/{id:int}/complete", async (
    int id,
    CompleteOrderService completeOrderService,
    CancellationToken cancellationToken) =>
{
    // endpoint は HTTP と use case の接続に集中します。
    var result = await completeOrderService.CompleteAsync(id, cancellationToken);

    return result switch
    {
        CompleteOrderResult.Completed => Results.NoContent(),
        CompleteOrderResult.NotFound => Results.NotFound(),
        CompleteOrderResult.AlreadyCanceled => Results.Conflict("キャンセル済みの注文は完了できません。"),
        _ => Results.StatusCode(StatusCodes.Status500InternalServerError)
    };
});

public class CompleteOrderService
{
    private readonly IOrderRepository orderRepository;
    private readonly TimeProvider timeProvider;

    public CompleteOrderService(IOrderRepository orderRepository, TimeProvider timeProvider)
    {
        this.orderRepository = orderRepository;
        this.timeProvider = timeProvider;
    }

    public async Task<CompleteOrderResult> CompleteAsync(int orderId, CancellationToken cancellationToken)
    {
        var order = await orderRepository.FindByIdAsync(orderId, cancellationToken);
        if (order is null)
        {
            return CompleteOrderResult.NotFound;
        }

        var completedAt = timeProvider.GetUtcNow();
        var result = order.Complete(completedAt);
        if (result != CompleteOrderResult.Completed)
        {
            return result;
        }

        await orderRepository.SaveAsync(order, cancellationToken);
        return CompleteOrderResult.Completed;
    }
}

public class Order
{
    public OrderStatus Status { get; private set; }
    public DateTimeOffset? CompletedAt { get; private set; }

    public CompleteOrderResult Complete(DateTimeOffset completedAt)
    {
        if (Status == OrderStatus.Canceled)
        {
            return CompleteOrderResult.AlreadyCanceled;
        }

        Status = OrderStatus.Completed;
        CompletedAt = completedAt;
        return CompleteOrderResult.Completed;
    }
}

public interface IOrderRepository
{
    Task<Order?> FindByIdAsync(int orderId, CancellationToken cancellationToken);
    Task SaveAsync(Order order, CancellationToken cancellationToken);
}

public enum OrderStatus
{
    Draft,
    Completed,
    Canceled
}

public enum CompleteOrderResult
{
    Completed,
    NotFound,
    AlreadyCanceled
}
```

## コードの読み方

悪い例では、endpoint の中に DB access、状態確認、状態変更、時刻取得が混ざっています。処理が小さいうちは動きますが、条件が増えると HTTP の入口を読むだけで業務ルールまで追う必要があります。

改善例では、endpoint は Application Service を呼ぶだけにしています。`CompleteOrderService` は use case の流れを管理し、`Order` は注文自身の業務ルールを持ち、`IOrderRepository` は保存方法を隠します。

## よくあるミス

- Controller や endpoint に業務判断を直接書く。
- Application Service がただ Repository を呼ぶだけの中継 class になっている。
- Domain object が getter / setter だけになり、業務ルールが外へ漏れている。
- Repository に在庫判断や状態遷移などの業務ルールを書く。
- すべての CRUD に重い層分けを適用し、単純な処理まで読みにくくする。

## 実務での使い方

新しい機能を作るときは、まず use case を1文で説明します。たとえば「注文を完了する」「支払いを確定する」「在庫を引き当てる」のように動詞で表せる処理は Application Service の候補です。

レビューでは、HTTP、業務ルール、永続化、外部 API 呼び出しがどこに書かれているかを確認します。層の名前よりも、変更理由が混ざっていないかを重視します。

## 練習問題

1. Controller に書かれた注文完了処理を Application Service へ移す。
2. `Order` に状態遷移の method を追加し、外部から `Status` を直接変更できないようにする。
3. Repository に入っている業務判断を Domain または Application Service へ移す。
