# DI の考え方

## 目的

Dependency Injection により、依存関係を外から渡す設計を理解します。

## 要点

- DI は依存オブジェクトをクラス内で作らず、外から受け取る考え方です。`new SmtpMailSender()` のように具体的な実装を直接作る代わりに、constructor で必要な依存を受け取ります。
- DI の目的は「interface を増やすこと」ではなく、依存方向を制御することです。業務ロジックが DB、メール送信、現在時刻、外部 API などの詳細に強く結び付くと、変更やテストが難しくなります。
- テストしやすく、実装を差し替えやすくなります。たとえば本番では SMTP でメールを送り、テストでは送信内容を記録する fake 実装に差し替える、といった構成ができます。
- ASP.NET Core では標準 DI コンテナーを使います。`builder.Services` に登録した型を framework が必要な場所へ渡してくれるため、Controller、Minimal API、Hosted Service などで同じ仕組みを使えます。
- Singleton、Scoped、Transient の lifetime は object の共有範囲を決めます。特に `DbContext` のような request 単位の依存を Singleton に入れると、不正な共有やスレッド安全性の問題につながります。
- 依存が多すぎる constructor は、クラスの責務が膨らんでいるサインです。DI で渡せるからといって何でも注入せず、業務上のまとまりごとに service を分割します。

## コード例

```csharp
// この例では「DI の考え方」の要点を最小のコードで確認します。
public class OrderService
{
    private readonly IMessageSender messageSender;

    public OrderService(IMessageSender messageSender)
    {
        this.messageSender = messageSender;
    }

    public void Complete() => messageSender.Send("completed");
}
```

## コードの読み方

このコード例は「DI の考え方」の基本形を確認するためのものです。上から順に、値や object を用意し、C# の構文や .NET API を使い、最後に結果を確認します。まず入力、処理、出力の 3 つに分けて読むと、初学者でも流れを追いやすくなります。

## 実務での使い方

DB、外部 API、メール送信、時計、ログなどを差し替え可能にします。DI は「interface を作ること」ではなく、依存方向を制御する設計です。

実装レベルの登録方法、lifetime、ASP.NET Core の標準 DI コンテナーについては [09_DIコンテナの実装.md](09_DIコンテナの実装.md) で扱います。

## よくあるミス

- DI コンテナーを Service Locator のようにどこからでも呼ぶ。
- 依存が多すぎるクラスを放置する。
- ライフタイム Singleton / Scoped / Transient を理解しない。

## レビュー観点

- クラス内で外部依存を直接 `new` していないか。
- 差し替えたい依存が constructor injection で渡されているか。
- 依存が多すぎる場合、クラスの責務が膨らんでいないか。
- lifetime の違いを説明できる登録になっているか。
