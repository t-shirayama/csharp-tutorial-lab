# 設計レビュー観点

## 目的

実装前後のレビューで、責務、依存方向、例外、ログ、設定、テスト容易性を横断して確認できるようにします。

## 前提

- [SOLID 原則](01_SOLID原則.md) を読んでいる
- [アプリケーションサービスと責務分割](14_アプリケーションサービスと責務分割.md) を読んでいる
- [DTO と Domain Model の使い分け](15_DTOとDomainModelの使い分け.md) を読んでいる

## 要点

- 設計レビューは、正解の形を押し付ける場ではなく、変更に耐えられるかを確認する場です。今の要件だけでなく、近い将来どこが変わりそうかを見ます。
- まず責務を確認します。Controller、Application Service、Domain、Repository、外部 API client が、それぞれの役割を越えていないかを見ます。
- 次に依存方向を確認します。業務ルールが DB、HTTP、外部 API、現在時刻などの詳細へ直接依存していると、テストや変更が難しくなります。
- 例外と validation は分類して確認します。利用者が直せる入力エラー、業務ルール違反、想定外のシステムエラーを同じ扱いにしないことが重要です。
- ログ、設定、secret は運用観点で確認します。障害調査に必要な ID がログにあり、secret や個人情報が出ず、必須設定が起動時に検証される形を目指します。
- テスト容易性は設計の結果です。外部依存を差し替えられるか、時刻や乱数を制御できるか、use case 単位でテストできるかを確認します。

## レビュー観点チェックリスト

| 観点 | 確認すること | よくある兆候 |
| --- | --- | --- |
| 責務 | 1つの class に複数の変更理由がないか | Service が巨大、Controller が長い |
| 依存方向 | Domain が Infrastructure に依存していないか | Domain で DbContext や HttpClient を使う |
| DTO | 外部契約と内部 model が分かれているか | Entity をそのまま API response にする |
| 例外 | validation、domain error、system error を分けているか | すべて 500、またはすべて例外 |
| ログ | 調査に必要な ID があり、secret が出ないか | message だけ、PII を出力 |
| 設定 | 必須設定を起動時に検証しているか | null のまま実行時に落ちる |
| DI | lifetime と依存数が妥当か | Singleton から Scoped に依存 |
| テスト | 外部依存を差し替えられるか | method 内で直接 `new` する |

## レビューコメント例

```text
この endpoint では在庫判定、注文状態の変更、DB 保存が同じ場所にあります。
HTTP の入口と業務ルールを分けるため、Application Service と Domain Model へ移せないか確認したいです。
```

```text
この API response は EF Core Entity をそのまま返しています。
内部項目が外へ出る可能性があるため、Response DTO に詰め替える形が安全です。
```

```text
この例外は利用者が入力を直せるケースに見えます。
未処理例外として 500 にするより、validation error として 400 で返す方が API 利用者に親切です。
```

## 設計レビューの進め方

1. まず use case を1文で説明する。
2. request から response までの流れを追う。
3. 業務判断がどこにあるか確認する。
4. 外部依存をどこで呼んでいるか確認する。
5. 失敗時の扱い、ログ、transaction 境界を確認する。
6. テストで差し替えたい依存が直接 `new` されていないか確認する。

## コードの読み方

レビューでは、個別の行だけでなく処理の流れを見ます。HTTP、use case、domain rule、DB access、外部 API、ログ、例外変換がどの層にあるかを線で追うと、責務の混ざり方が見つけやすくなります。

## よくあるミス

- SOLID や Clean Architecture の名前だけで指摘し、具体的な変更理由を説明しない。
- すべての小さな処理に重い設計を要求する。
- 動いているかだけを見て、失敗時、変更時、テスト時の扱いを見ない。
- レビュー観点が人によってばらつき、毎回違う基準になる。

## 実務での使い方

PR レビュー、設計相談、リファクタリング前の棚卸しで使います。最初から完璧な設計を求めるより、変更が入りやすい箇所、障害時に困る箇所、テストしづらい箇所を優先して直します。

## 練習問題

1. 既存の Controller を読み、業務判断が書かれている行を探す。
2. API response に Entity をそのまま返していないか確認する。
3. DI の lifetime と constructor の依存数をレビューする。
4. ログに出してよい値と出してはいけない値を分類する。
