# DI コンテナの実装

## 目的

ASP.NET Core の標準 DI コンテナーでサービスを登録し、lifetime を判断できるようにします。

## 前提

- [DI の考え方](02_DIの考え方.md) を読んでいる
- [設定管理](05_設定管理.md) を読んでいる
- `dotnet new webapi` の `Program.cs` を読める

## 要点

- DI 登録は `builder.Services` に集約します。
- constructor injection で依存を受け取ります。
- lifetime は Singleton、Scoped、Transient の違いを理解して選びます。
- ASP.NET Core の通常リクエスト処理では、DB や業務サービスは Scoped にすることが多いです。

## lifetime の使い分け

| lifetime | 生成タイミング | 主な用途 | 注意点 |
| --- | --- | --- | --- |
| Singleton | アプリ全体で1つ | stateless な共通機能、キャッシュ、設定参照 | Scoped サービスに依存しない |
| Scoped | リクエストごとに1つ | DbContext、業務サービス、リクエスト単位の処理 | バックグラウンド処理では scope を作る |
| Transient | 解決ごとに新規 | 軽量で状態を持たない処理 | 依存が多いと生成コストが増える |

## コード例

`Program.cs` で interface と実装を登録します。

```csharp
// この例では「DI コンテナの実装」の要点を最小のコードで確認します。
var builder = WebApplication.CreateBuilder(args);

// リクエストごとに OrderService を1つ作ります。
builder.Services.AddScoped<IOrderService, OrderService>();

// SystemClock は状態を持たないため、アプリ全体で1つ共有します。
builder.Services.AddSingleton<IClock, SystemClock>();

var app = builder.Build();

app.MapPost("/orders/{id:int}/complete", (int id, IOrderService orderService) =>
{
    // endpoint は DI から受け取った service に業務処理を委譲します。
    orderService.Complete(id);
    return Results.NoContent();
});

app.Run();

public interface IOrderService
{
    void Complete(int orderId);
}

public class OrderService : IOrderService
{
    private readonly IClock clock;

    // constructor injection で、時刻取得の依存を外から受け取ります。
    public OrderService(IClock clock)
    {
        this.clock = clock;
    }

    public void Complete(int orderId)
    {
        Console.WriteLine($"Order {orderId} completed at {clock.Now}.");
    }
}

public interface IClock
{
    DateTimeOffset Now { get; }
}

public class SystemClock : IClock
{
    // 実行時は現在の UTC 時刻を返します。テストでは別実装に差し替えます。
    public DateTimeOffset Now => DateTimeOffset.UtcNow;
}
```

## コードの読み方

`builder.Services.AddScoped<IOrderService, OrderService>()` により、endpoint で `IOrderService` が必要になったとき `OrderService` が渡されます。`OrderService` は `IClock` に依存しているため、DI コンテナーは `SystemClock` も解決します。

endpoint は業務処理を直接書かず、`orderService.Complete(id)` に委譲しています。これにより、Web の入口と業務ロジックを分けてテストしやすくできます。

## 避けるべき実装

```csharp
// この例では「DI コンテナの実装」の要点を最小のコードで確認します。
public class OrderService
{
    private readonly SystemClock clock = new();
}
```

クラス内で依存を直接 `new` すると、テスト時に差し替えにくくなります。

## 実務での使い方

Controller、endpoint、service、repository、外部 API client を DI でつなぎます。登録は feature ごとに extension method へ分けると `Program.cs` が肥大化しにくくなります。

```csharp
// この例では「DI コンテナの実装」の要点を最小のコードで確認します。
public static class OrderServiceCollectionExtensions
{
    public static IServiceCollection AddOrderFeature(this IServiceCollection services)
    {
        // Order 機能に必要な DI 登録をここへ集約します。
        services.AddScoped<IOrderService, OrderService>();
        return services;
    }
}
```

## よくあるミス

- Singleton から Scoped の `DbContext` に依存する。
- DI コンテナーを任意の場所で呼び出す Service Locator にする。
- interface を機械的に作りすぎる。
- 登録漏れを起動時ではなく実行時まで見逃す。

## レビュー観点

- lifetime の理由を説明できるか。
- 依存が constructor injection に集約されているか。
- `Program.cs` が登録処理で読みづらくなっていないか。
- service の責務が大きくなり、依存が増えすぎていないか。
