# 型設計の選び方

## 目的

`class`、`record`、`record struct`、`struct`、`enum` を、用途と変更しやすさに合わせて選べるようにします。

## 前提

- [class と object](../02_オブジェクト指向/01_classとobject.md) を読んでいる
- [record と値の等価性](09_recordと値の等価性.md) を読んでいる
- [required と init](08_requiredとinit.md) を読んでいる

## 要点

- `class` は参照型で、identity や lifecycle を持つ object に向いています。注文、ユーザー、集約 root、状態を変更しながら扱う service などは `class` が自然です。
- `record` は値の等価性を持つ参照型です。API response、query result、設定値、message payload のように、同じ値なら同じものとして扱いたいデータに向いています。
- `struct` は値型です。小さく、不変で、頻繁に大量生成される値に向いていますが、copy の挙動や boxing を理解せずに使うと逆に分かりにくくなります。
- `record struct` は値型と record の機能を組み合わせます。小さな値オブジェクトで値の等価性を簡潔に表したい場合に候補になります。
- `enum` は固定された選択肢を表します。状態、種別、区分を表すのに便利ですが、値ごとに振る舞いが増える場合は strategy や polymorphism も検討します。
- 型選択では「何を表すか」「変更されるか」「等価性を何で判断するか」「外部へ公開する契約か」「永続化や serializer と相性がよいか」を確認します。

## 選び方の目安

| 型 | 向いているもの | 注意点 |
| --- | --- | --- |
| `class` | identity、状態変更、service、domain entity | mutable にしすぎると不正状態を作りやすい |
| `record` | DTO、response、query result、設定 snapshot | entity に使うと identity と値の等価性が混ざることがある |
| `struct` | 小さな値、低レイヤー API、性能が必要な場面 | copy、boxing、default 値に注意する |
| `record struct` | 小さな値オブジェクト、座標、範囲、数量 | 大きい値や可変値には向かない |
| `enum` | 状態、種別、固定区分 | 値ごとの処理が増えたら分岐が肥大化する |

## class を選ぶ例

```csharp
public class Order
{
    private readonly List<OrderLine> lines = new();

    public int Id { get; }
    public OrderStatus Status { get; private set; } = OrderStatus.Draft;
    public IReadOnlyList<OrderLine> Lines => lines;

    public Order(int id)
    {
        Id = id;
    }

    public void AddLine(string productCode, int quantity)
    {
        if (Status != OrderStatus.Draft)
        {
            throw new InvalidOperationException("確定済みの注文は変更できません。");
        }

        lines.Add(new OrderLine(productCode, quantity));
    }
}

public record OrderLine(string ProductCode, int Quantity);

public enum OrderStatus
{
    Draft,
    Completed,
    Canceled
}
```

## record を選ぶ例

```csharp
// API response は、外部へ返す値の形を表すだけなので record と相性がよいです。
public record OrderResponse(int Id, string Status, decimal TotalAmount);

var response = new OrderResponse(123, "Completed", 12000m);
Console.WriteLine(response);
```

## record struct を選ぶ例

```csharp
public readonly record struct Money(decimal Amount, string Currency)
{
    public Money Add(Money other)
    {
        if (Currency != other.Currency)
        {
            throw new InvalidOperationException("通貨が異なる金額は加算できません。");
        }

        return new Money(Amount + other.Amount, Currency);
    }
}

var total = new Money(1000m, "JPY").Add(new Money(500m, "JPY"));
Console.WriteLine(total.Amount);
```

## コードの読み方

`Order` は identity と状態変更を持つため `class` にしています。`AddLine` の中に業務ルールを置くことで、外部から自由に明細を変更できないようにしています。

`OrderResponse` は API の戻り値なので、値を運ぶだけの `record` として表しています。`Money` は小さな値で、同じ金額と通貨なら同じ値として扱いたいため `readonly record struct` にしています。

## よくあるミス

- DTO、Domain Model、DB Entity をすべて同じ型にする。
- identity が重要な entity を record にして、値の等価性と混同する。
- 大きく可変なデータを struct にして、copy の挙動で混乱する。
- enum の値ごとの処理が増えても、巨大な `switch` のまま放置する。
- `required` や `init` を付けただけで、業務上不正な値を防げたと思う。

## 実務での使い方

API 境界では DTO / response として `record`、業務ルールを持つ中心では `class`、小さな値オブジェクトでは `record struct` を候補にします。EF Core entity は framework の制約や変更追跡との相性もあるため、record を使う場合は慎重に確認します。

型を選ぶときは、後から変更される理由を考えます。外部 API の契約変更、DB schema 変更、業務ルール変更を同じ型に背負わせると、変更の影響範囲が広がります。
