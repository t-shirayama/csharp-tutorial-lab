# 共変性と反変性

## 目的

generic interface や delegate で出てくる covariance / contravariance の入口を理解します。

## 前提

- [generics](02_generics.md) を読んでいる
- [継承とポリモーフィズム](../02_オブジェクト指向/06_継承とポリモーフィズム.md) を読んでいる

## 要点

- 共変性は、より派生した型を返すものを基底型として扱える性質です。
- 反変性は、より基底の型を受け取るものを派生型用として扱える性質です。
- C# では `out` と `in` で variant type parameter を表します。

## コード例

```csharp
// この例では「共変性と反変性」の要点を最小のコードで確認します。
IEnumerable<string> strings = new List<string> { "A", "B" };

// IEnumerable<out T> は共変なので、IEnumerable<object> として扱えます。
IEnumerable<object> objects = strings;

foreach (var value in objects)
{
    Console.WriteLine(value);
}
```

## 反変性の例

```csharp
// この例では「共変性と反変性」の要点を最小のコードで確認します。
Action<object> writeObject = value => Console.WriteLine(value);

// Action<in T> は反変なので、Action<string> として扱えます。
Action<string> writeString = writeObject;
writeString("hello");
```

## コードの読み方

`IEnumerable<T>` は値を取り出すだけなので、`string` の列挙を `object` の列挙として見ても安全です。`Action<T>` は値を受け取るだけなので、`object` を受け取れる処理は `string` も受け取れます。

## 実務での使い方

`IEnumerable<T>`、`IReadOnlyList<T>`、`Func<T>`、`Action<T>`、event handler、interface 設計で見かけます。自作 generic interface で `in` / `out` を付ける場合は、その型 parameter が入力だけか出力だけかを確認します。

## よくあるミス

- `List<string>` を `List<object>` に代入できると思う。
- `in` と `out` を nullable の入出力と混同する。
- 入力にも出力にも使う type parameter に variance を付けようとする。

## 練習問題

1. `IEnumerable<string>` を `IEnumerable<object>` に代入する。
2. `List<string>` を `List<object>` に代入できないことを確認する。
3. `Func<string>` と `Func<object>` の代入関係を確認する。
