# 不変コレクション

## 目的

`ImmutableArray<T>` や `ImmutableList<T>` の考え方と、通常の collection との違いを理解します。

## 前提

- [配列と List](01_配列とList.md) を読んでいる
- [record](../02_オブジェクト指向/04_record.md) を読んでいる

## 要点

- 不変コレクションは、作成後に元の instance を変更しません。
- 変更操作は、新しい collection を返します。
- 共有状態を減らしたい場面で役立ちます。

## コード例

```csharp
// この例では「不変コレクション」の要点を最小のコードで確認します。
using System.Collections.Immutable;

var original = ImmutableArray.Create("A", "B");
var added = original.Add("C");

Console.WriteLine(original.Length); // 2
Console.WriteLine(added.Length);    // 3
```

## コードの読み方

`original.Add("C")` は `original` 自体を変更しません。追加後の値は `added` に入ります。record や immutable な設計と相性がよいです。

## 実務での使い方

設定値、読み取り専用の snapshot、並行処理で共有する値、ドメインモデルの状態表現で使います。頻繁な追加・削除がある hot path では、通常の mutable collection の方が適切な場合もあります。

## よくあるミス

- `Add` が元の collection を変更すると思い込む。
- 大量更新に不変コレクションを使い、性能問題にする。
- 読み取り専用 view と不変 collection を混同する。
