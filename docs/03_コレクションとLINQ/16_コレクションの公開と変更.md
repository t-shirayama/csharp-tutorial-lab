# コレクションの公開と変更

## 目的

`List<T>` をそのまま外部へ公開する危険性、読み取り専用 interface、不変コレクション、列挙中の変更を理解します。

## 前提

- [配列と List](01_配列とList.md) を読んでいる
- [不変コレクション](12_不変コレクション.md) を読んでいる

## 要点

- `List<T>` は変更しやすい collection です。便利な一方で、外部にそのまま渡すと呼び出し側が `Add`、`Remove`、`Clear` できてしまい、クラス内の状態を壊される可能性があります。
- 外部に「読み取りだけ」を許したい場合は、戻り値を `IReadOnlyList<T>` や `IReadOnlyCollection<T>` にします。呼び出し側へ、変更してよい collection ではないことを型で伝えられます。
- `IEnumerable<T>` は「列挙できる」ことだけを表します。件数や index access が必要なら `IReadOnlyCollection<T>` や `IReadOnlyList<T>` の方が意図が明確です。
- `ReadOnlyCollection<T>` や `AsReadOnly()` は読み取り専用 view です。元の `List<T>` が変更されると view の内容も変わるため、完全な不変とは違います。
- `ImmutableArray<T>` や `ImmutableList<T>` は変更操作で新しい instance を返します。共有状態を減らしたい場合に有効ですが、大量更新の hot path では性能を確認します。
- `foreach` で列挙中の `List<T>` を変更すると `InvalidOperationException` になることがあります。削除したい場合は `RemoveAll`、逆順 for、または新しい list への絞り込みを使います。

## 悪い例

```csharp
public class Order
{
    private readonly List<OrderLine> lines = new();

    // List<T> をそのまま返すと、呼び出し側が自由に変更できてしまいます。
    public List<OrderLine> Lines => lines;

    public void AddLine(string productCode, int quantity)
    {
        lines.Add(new OrderLine(productCode, quantity));
    }
}

var order = new Order();
order.AddLine("P-001", 2);

// クラスのルールを通らずに、外部から明細を消せてしまいます。
order.Lines.Clear();

public record OrderLine(string ProductCode, int Quantity);
```

## 改善例

```csharp
public class Order
{
    private readonly List<OrderLine> lines = new();

    // 外部には読み取り専用として公開します。
    // 内部では List<T> のままなので、Order クラスの method 経由で変更できます。
    public IReadOnlyList<OrderLine> Lines => lines;

    public void AddLine(string productCode, int quantity)
    {
        if (quantity <= 0)
        {
            throw new ArgumentOutOfRangeException(nameof(quantity), "数量は1以上にします。");
        }

        lines.Add(new OrderLine(productCode, quantity));
    }

    public void RemoveLine(string productCode)
    {
        lines.RemoveAll(line => line.ProductCode == productCode);
    }
}

public record OrderLine(string ProductCode, int Quantity);
```

## 列挙中の変更

```csharp
var numbers = new List<int> { 1, 2, 3, 4, 5 };

// foreach 中に Remove すると InvalidOperationException になる可能性があります。
// 削除条件が明確なら RemoveAll を使うと意図も安全性も分かりやすくなります。
numbers.RemoveAll(number => number % 2 == 0);

Console.WriteLine(string.Join(",", numbers)); // 1,3,5
```

新しい list を作る方法もあります。

```csharp
var numbers = new List<int> { 1, 2, 3, 4, 5 };

// 元の list を変更せず、条件に合う要素だけで新しい list を作ります。
var oddNumbers = numbers
    .Where(number => number % 2 != 0)
    .ToList();

Console.WriteLine(string.Join(",", oddNumbers)); // 1,3,5
```

## コードの読み方

悪い例では、`Lines` の型が `List<OrderLine>` なので、呼び出し側が `Clear()` できます。`Order` クラスの `AddLine` に入力チェックを書いても、外部から直接変更できるならルールを守れません。

改善例では、内部状態は `List<T>` で持ち、公開する型だけを `IReadOnlyList<T>` にしています。変更は `AddLine` や `RemoveLine` という業務上の意味を持つ method に集めます。

## よくあるミス

- property で `List<T>` をそのまま公開する。
- `IReadOnlyList<T>` にしただけで完全に不変になったと思い込む。
- `AsReadOnly()` の view と immutable collection を混同する。
- `foreach` 中に `Add` や `Remove` をして例外にする。
- 呼び出し側が件数や index を必要としているのに、戻り値を何でも `IEnumerable<T>` にする。

## 実務での使い方

ドメインモデル、DTO、設定値、検索結果、API response で collection を返すときに判断します。内部で変更する必要があるなら `List<T>`、外部には `IReadOnlyList<T>`、共有状態を避けたいなら immutable collection、単に順に処理できればよいなら `IEnumerable<T>` が候補になります。

レビューでは、collection の所有者が誰かを確認します。呼び出し側が変更してよい collection なのか、class 内の method だけが変更してよい状態なのかを型と method 名で表します。

## 練習問題

1. `List<T>` を公開している property を `IReadOnlyList<T>` に変更する。
2. `foreach` 中に削除している処理を `RemoveAll` に書き換える。
3. `IEnumerable<T>`、`IReadOnlyList<T>`、`ImmutableArray<T>` のどれを返すべきか、3つのケースで判断する。
