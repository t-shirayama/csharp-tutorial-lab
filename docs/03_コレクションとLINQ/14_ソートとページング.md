# ソートとページング

## 目的

一覧を並び替え、必要な件数だけ取り出す `OrderBy`、`ThenBy`、`Skip`、`Take`、`Chunk` の使いどころを理解します。

## 前提

- [LINQ の基本](04_LINQの基本.md) を読んでいる
- [LINQ の性能と ToList](09_LINQの性能とToList.md) を読んでいる

## 要点

- `OrderBy` は最初の並び替え条件、`ThenBy` は同じ値だった場合の追加条件です。名前順だけでなく、日付の降順、優先度、ID などを組み合わせて、結果の順序が毎回安定するようにします。
- `OrderByDescending` と `ThenByDescending` は降順に並べます。新しいものを先に出す、価格が高い順に出す、スコアが高い順に出す、といった画面一覧でよく使います。
- `Skip` と `Take` は page 表示で使います。`Skip((page - 1) * pageSize).Take(pageSize)` の形が基本ですが、`page` と `pageSize` は 0 以下や大きすぎる値を受けないように検証します。
- DB query では、`Skip` / `Take` の前に `OrderBy` を指定します。並び順がない page は、同じ request でも結果が揺れる可能性があり、利用者にとって扱いにくくなります。
- `Chunk` は一定件数ごとのまとまりに分割します。大量データを一括処理せず、100 件ずつ送信する、画面表示用に行単位で分ける、といった場面で役立ちます。
- メモリ上の LINQ と EF Core query では性能の見方が違います。EF Core では `OrderBy`、`Skip`、`Take` が SQL に変換されるか、どの列に index があるかも確認します。

## コード例

```csharp
// 一覧画面に表示する商品データを用意します。
var products = new[]
{
    new Product("P-001", "Keyboard", "Device", 12000m, new DateTime(2026, 5, 1)),
    new Product("P-002", "Mouse", "Device", 4500m, new DateTime(2026, 5, 5)),
    new Product("P-003", "Notebook", "Stationery", 700m, new DateTime(2026, 4, 20)),
    new Product("P-004", "Monitor", "Device", 32000m, new DateTime(2026, 5, 3))
};

var page = 1;
var pageSize = 2;

// まずカテゴリ、次に価格の高い順、最後にコード順で並べます。
// 最後に一意に近い条件を入れると、同じ条件のデータでも順序が安定します。
var displayProducts = products
    .OrderBy(product => product.Category)
    .ThenByDescending(product => product.Price)
    .ThenBy(product => product.Code)
    .Skip((page - 1) * pageSize)
    .Take(pageSize)
    .Select(product => new ProductListItem(
        product.Code,
        product.Name,
        $"{product.Price:N0}円"))
    .ToList();

foreach (var product in displayProducts)
{
    Console.WriteLine($"{product.Code}: {product.Name} / {product.PriceLabel}");
}

public record Product(string Code, string Name, string Category, decimal Price, DateTime CreatedAt);
public record ProductListItem(string Code, string Name, string PriceLabel);
```

## Chunk の例

```csharp
var productCodes = Enumerable.Range(1, 250)
    .Select(number => $"P-{number:000}")
    .ToList();

// 外部 API へ一度に送りすぎないように、100 件ずつに分けます。
foreach (var batch in productCodes.Chunk(100))
{
    Console.WriteLine($"送信件数: {batch.Length}");
}
```

## コードの読み方

最初の例では、`OrderBy` と `ThenBy` で並び順を決めてから、`Skip` と `Take` で指定 page の分だけ取り出しています。`Select` は画面表示用の形に変換する処理です。実務では、並び替え、page 切り出し、表示用変換の順序を意識すると読みやすくなります。

`Chunk` の例では、250 件のコードを 100 件ずつに分けています。`Chunk` が返す各要素は配列なので、batch ごとの件数は `Length` で確認できます。

## よくあるミス

- `Skip` / `Take` の前に `OrderBy` を書かず、page の結果が安定しない。
- `page` が 0 のときに `Skip` の計算が負数になる。
- `pageSize` に上限を設けず、大量件数を一度に返してしまう。
- `OrderBy` を何度も書いて、前の並び替え条件を上書きしてしまう。2つ目以降は `ThenBy` を使います。
- DB で page 化すべき処理を、先に全件 `ToList()` してからメモリ上で `Skip` / `Take` する。

## 実務での使い方

一覧画面、検索 API、管理画面、CSV 出力のプレビューなどで使います。外部から sort 条件を受け取る場合は、許可する列名だけを mapping し、任意の文字列をそのまま query に使わないようにします。

EF Core では、page 化の前に DB 側で絞り込み、並び替え、件数制限が行われるように query を組みます。大量データでは、offset paging だけでなく keyset pagination を検討することもあります。
